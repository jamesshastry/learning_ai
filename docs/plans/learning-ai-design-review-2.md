VERDICT: NEEDS_REVISION

## Summary Assessment

The v2 revision substantively addressed all five critical issues from round 1, and the design is significantly more implementable. However, the revision introduced two new critical issues: the combined Claude API response parsing strategy is fragile and underspecified, and there is a contradictory dependency between the Phase 1 CLI surface and the pipeline it relies on.

## Critical Issues (must fix)

### 1. Combined Claude API response parsing is fragile and unspecified

The design says the single `analyze` call instructs Claude to return "a response with two clearly delimited sections" (structured notes as markdown, concepts as YAML), and then "the analyze stage parses the response, splits it into the two outputs." But it never specifies what the delimiter is, how the parser identifies where markdown ends and YAML begins, or what happens when parsing fails.

This is the core of the entire pipeline and it is the least specified part of the design.

Real problems that will occur:
- Claude may wrap the YAML in a markdown code fence (` ```yaml ... ``` `) or it may not, depending on the model, temperature, and prompt phrasing.
- The notes markdown itself could contain YAML code fences (e.g., in a lecture about configuration management), breaking a naive fence-based delimiter.
- If parsing fails, do you retry the API call ($0.19 wasted), fall back to regex extraction, or mark as error?

**Fix:** Specify the exact delimiter protocol. The most reliable approach: instruct Claude to output the YAML block inside a uniquely tagged fence (e.g., ```` ```concepts_yaml ```` / ```` ``` ````) and parse using that specific tag. Define the fallback behavior when parsing fails (retry once with a stricter prompt, then mark as error). Consider whether using Claude's tool_use / structured output mode would be more reliable than free-form text parsing — the concepts YAML is essentially structured data and tool_use guarantees valid JSON output that can be converted to YAML.

### 2. Phase 1 CLI includes `learn process` but not the `ingest` pipeline it depends on

Phase 1 says `learn process <youtube-url> --course cs153` is the on-demand single-video command. But the pipeline stages (resolve, detect duration, check cache, transcribe, normalize, analyze, write) are the same stages used by `learn ingest`. The design describes these stages under "Pipeline Architecture" but then says `learn ingest` is Phase 2.

This creates a confusing question: does `learn process` implement the full 7-stage pipeline for a single video? If so, it is essentially `learn ingest` for one video, and the Phase 2 `learn ingest` only adds playlist iteration, concurrency, and resume-on-failure. That is a reasonable split, but the design does not state this explicitly. The pipeline stages section describes them in the context of `ingest`, not `process`.

Separately, the `learn add` command in Phase 1 "resolves video IDs" from a playlist URL and writes `course.yaml`. But the playlist resolution logic (stage 1 of the pipeline: scraping `ytInitialData`) is only described in the pipeline architecture section, which is framed around `ingest`. It is unclear whether `learn add` does the full playlist resolution in Phase 1, or just stores the playlist URL and defers resolution to Phase 2.

**Fix:** Explicitly state that `learn process` runs the full per-lecture pipeline (stages 2-7) for a single video URL. State that `learn add` resolves the playlist to extract all video IDs and populates `course.yaml` with `pending` entries in Phase 1. State that Phase 2's `learn ingest` reuses the same per-lecture pipeline but adds iteration over pending lectures, concurrency control, and resume logic.

## Suggestions (nice to have)

### 1. Merge algorithm does not handle the "existing concepts" list for Claude

The analyze prompt includes "a list of existing concept names from `knowledge-graph/concepts/`" so Claude can reference existing concepts. But in Phase 1, concept files in `knowledge-graph/concepts/` do not exist yet — they are generated by `learn graph build` in Phase 3. The only concepts that exist at ingestion time are in the per-lecture `concepts.yaml` files.

This means the "existing concepts" list passed to Claude must be assembled by scanning all `concepts.yaml` files across all courses, not by reading `knowledge-graph/concepts/`. The design should clarify this. In Phase 1, this list will grow organically as each lecture is processed, which is fine. But the source of truth for the list needs to be specified: is it the union of all `concepts.yaml` files, or the `knowledge-graph/concepts/` directory, or a separate index?

For Phase 1 simplicity, consider maintaining a flat file like `knowledge-graph/concept-index.txt` (one concept name per line) that is appended after each lecture's analysis. This avoids scanning every `concepts.yaml` on every invocation.

### 2. Levenshtein distance threshold of 3 is too aggressive for short concept names

The merge algorithm's Pass 2 flags concepts with Levenshtein distance <= 3 as near-duplicates. For short names, this will produce excessive false positives: "RNN" and "CNN" have distance 1, "CUDA" and "CUDA cores" have distance 6 (so it would miss this real near-duplicate), while "GPU" and "TPU" have distance 1 (flagged as near-duplicate despite being distinct concepts).

Consider using normalized edit distance (edit distance / max length) with a threshold like 0.2-0.3, or Jaro-Winkler similarity, which handles abbreviations better. Alternatively, keep Levenshtein but only for names above a minimum length (e.g., 8+ characters).

### 3. Rate limit tracking location may cause issues

The design stores rate limit tracking in `~/.learning-ai/usage.json`. This is a global file outside the project. If the user runs two separate learning-ai projects or two terminal sessions simultaneously, they could exceed the rate limit because the file is not locked. For Phase 1 (single-video processing) this is unlikely to matter, but for Phase 2 (batch ingestion with concurrency), concurrent access to this file needs to be addressed. Consider advisory file locking or an atomic read-modify-write pattern.

### 4. VTT deduplication for rolling captions needs specification

The normalize stage handles yt-dlp VTT output by "deduplicating rolling captions." YouTube's auto-generated VTT uses a rolling window where each cue contains the previous line plus a new line, resulting in massive duplication. The deduplication algorithm is important for transcript quality but is not specified. A common approach: track the last N characters of emitted text, and for each new cue, only emit the suffix that was not already seen. This is worth specifying because getting it wrong means every sentence appears 2-3 times in the transcript, which would degrade Claude's analysis.

### 5. `learn notes cs153 1` needs clarification on how "1" maps to a lecture

The `learn notes cs153 1` command uses a positional number. But lecture IDs in `course.yaml` are strings (e.g., `"01"`). Does `1` mean the first lecture in the YAML array (array index), or the lecture with `id: "01"` (string match), or the lecture with `id: "1"` (numeric match)? This is a minor UX detail but will cause confusion if not specified. Recommend matching on the `id` field with flexible parsing (strip leading zeros).

### 6. No specification for what `learn status` actually displays

Phase 1 includes `learn status` but the design never shows example output. What does it print? A table of lectures with their status? Counts of pending/completed/error? Rate limit usage? Knowing the output format helps both the implementer and the user. A one-line example would suffice.

## Verified Claims (things I confirmed are correct)

### Round 1 Critical Issue #1 (Fallback transcription): RESOLVED

The v2 design adds yt-dlp auto-captions as a degraded fallback with clear trigger conditions (duration > 90 min, service error, rate limit exhausted). The `transcript_source` field tracks provenance. The yt-dlp command shown (`--write-auto-sub --sub-lang en --skip-download --sub-format vtt`) is correct for extracting auto-generated captions. The normalize stage ensures downstream stages are source-agnostic. The limitations of fallback captions are honestly documented.

### Round 1 Critical Issue #2 (Concept extraction in Phase 1): RESOLVED

Concept extraction is now part of the combined `analyze` stage in Phase 1. The single-call strategy eliminates the rework problem (no need to reprocess lectures when Phase 3 ships) and halves API costs. The `concepts.yaml` file is written at ingestion time. Phase 3 only aggregates and visualizes already-extracted concepts.

### Round 1 Critical Issue #3 (Knowledge graph merge algorithm): RESOLVED

The merge algorithm is now concrete with two passes (exact matching, then ambiguity detection), specific merge rules for each field (name: keep longer, aliases: union, definition: keep longer, tags: union, sources: union, relations: union, first_seen: keep earliest), and a `merge-log.yaml` for ambiguous cases. This is implementable as specified. The fuzzy merge (Claude-assisted) is correctly deferred to Phase 3.

### Round 1 Critical Issue #4 (Videos over 90 minutes): RESOLVED

Duration detection is now an explicit pipeline stage. The implementation approach (parsing `ytInitialPlayerResponse` from YouTube page HTML) is correct — this JSON object is reliably present in server-rendered YouTube pages. The decision logic (<=5400s: primary, >5400s: fallback, unavailable: try primary then fallback) is sound.

### Round 1 Critical Issue #5 (Cached transcript permalink): RESOLVED

The design now explicitly states that the fetch URL is constructed from the `/api/check` response's `permalink` field. Error handling includes explicit redirect following as a retry strategy. This was a subtle bug that would have caused silent failures.

### Round 1 Suggestions incorporated correctly

- Phase 1 CLI reduced from 7+ to 4 commands: confirmed.
- `transcript.txt` instead of `transcript.md`: confirmed.
- `learn ask` rerouted through Claude API: confirmed, correctly moved to Phase 2.
- Combined prompt: confirmed, merged with critical fix #2.
- `schema_version` added to `course.yaml`: confirmed at top level.

### Cost estimate is reasonable

The $0.19/lecture estimate for a single combined call is plausible. A 75-minute lecture at ~150 words/minute produces ~11,250 words (~15K input tokens). With Claude Sonnet 4.6 pricing at ~$3/M input + $15/M output, and assuming ~2K output tokens for notes + concepts YAML, the per-lecture cost is approximately (15K * $3/M) + (2K * $15/M) = $0.045 + $0.030 = ~$0.075. The design's $0.19 estimate is conservative (likely accounting for prompt template overhead and safety margin), which is fine for planning purposes.

### Phasing is now logically sound

The dependency chain is correct: Phase 1 extracts concepts at ingestion time, Phase 2 adds batch processing, Phase 3 aggregates and visualizes. No phase depends on work that has not been done in a prior phase (with the caveat in Critical Issue #2 above about `learn process` needing the pipeline).
